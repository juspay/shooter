# Universal Installer & Full Lifecycle CLI

**Date:** 2026-03-26
**Status:** Approved

## Goal

Make Shooter installable and runnable on any macOS machine with a single command. Zero prerequisites beyond a working terminal (and Xcode Command Line Tools for native module compilation). The user never needs to know about pnpm, launchctl, tsx, or project internals.

## Prerequisites

| Requirement | Why | How checked |
|-------------|-----|-------------|
| Xcode Command Line Tools | `node-pty` and `better-sqlite3` require C++ compilation via node-gyp | `xcode-select -p` in install.sh; offer `xcode-select --install` if missing |

This is the only thing the installer cannot do silently ‚Äî Apple requires interactive consent for Xcode CLT. The installer detects it early and guides the user through the one-time install before proceeding.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Package manager | pnpm internally, invisible to user | Reproducible builds via lockfile; user never types `pnpm` |
| Node installation | Detect existing, `brew install node` as fallback | Respect what's there, don't impose a version manager |
| Launcher wrapper | Hardcoded fast path + self-healing fallback chain | Fast startup, no stale paths after Node upgrades |
| CLI scope | Full lifecycle (start/stop/restart/status/update/logs/autostart) | User never needs to touch internals |
| Entry points | All three: `curl\|sh`, `npx`, `brew install` | Different doors, same room |

## Installation Entry Points

Three ways in, ordered by user knowledge required:

### 1. `curl | sh` (Zero knowledge)

```bash
curl -fsSL https://raw.githubusercontent.com/juspay/shooter/release/scripts/install.sh | sh
```

Flow:
1. Detect OS (macOS only for now, Linux later)
2. Find Node via detection chain (see below)
3. Check Xcode CLT (`xcode-select -p`); prompt to install if missing
4. Bootstrap pnpm: try `corepack enable` first; if it fails (permissions), fall back to `npm install -g pnpm` or use `node_modules/.bin/pnpm` after install
5. Clone repo to `~/.shooter/repo` (code lives here, separate from data)
6. `pnpm install` (now available via corepack or fallback)
7. Hand off to `node scripts/setup.cjs` (interactive wizard ‚Äî writes `.env` to `~/.shooter/`)
8. `pnpm build`
9. Offer auto-start (launchd plist + generated wrapper script)
10. Link `shooter` CLI globally to user-writable PATH (`~/.local/bin/` or `~/bin/`)

### 2. `npx` (Has Node)

```bash
npx @juspay/shooter setup
```

Skips Node detection (already exists). The `npx` invocation downloads the package to a temp directory, but `setup` in this context acts as a **thin bootstrapper**: it detects whether `~/.shooter/repo` exists, clones it if not, then hands off to the repo's own `scripts/setup.cjs`. This way the same setup.cjs logic runs regardless of entry point, and ROOT resolution always points at the cloned repo, not an ephemeral npx directory.

### 3. `brew install` (Homebrew user)

```bash
brew tap juspay/shooter
brew install shooter
```

Homebrew formula declares `depends_on "node@22"`. Install block runs pnpm install + build. Links the `shooter` CLI. User runs `shooter setup` to configure.

All three converge on the same `setup.cjs` wizard and the same `shooter` CLI.

## Directory Layout

`~/.shooter` is the top-level home. Code and data are separated to prevent fresh installs from destroying runtime state:

```
~/.shooter/
‚îú‚îÄ‚îÄ repo/                  # git clone lives here (code)
‚îÇ   ‚îú‚îÄ‚îÄ build/
‚îÇ   ‚îú‚îÄ‚îÄ node_modules/
‚îÇ   ‚îú‚îÄ‚îÄ server.ts
‚îÇ   ‚îú‚îÄ‚îÄ bin/shooter.cjs
‚îÇ   ‚îî‚îÄ‚îÄ ...
‚îú‚îÄ‚îÄ shooter.db             # SQLite database (data ‚Äî preserved across updates)
‚îú‚îÄ‚îÄ device-tokens.json     # registered devices (data)
‚îú‚îÄ‚îÄ .env                   # configuration (data ‚Äî preserved across updates)
‚îú‚îÄ‚îÄ launcher.sh            # generated by `shooter autostart on`
‚îú‚îÄ‚îÄ shooter.pid            # PID file for stop/status
‚îî‚îÄ‚îÄ logs/                  # alternative log location if not using ~/Library/Logs
```

**Key invariant:** `shooter update` and fresh installs only touch `~/.shooter/repo/`. The `.env`, `shooter.db`, and `device-tokens.json` files live one level up and are never deleted by any automated operation.

`setup.cjs` writes `.env` to `~/.shooter/.env`. `server.ts` loads it via `dotenv` with an explicit path. The repo's `.env.example` stays as documentation only.

## Node Detection Chain

Checked in sequence, first match wins:

| Priority | Check | How | Notes |
|----------|-------|-----|-------|
| 1 | `node` on PATH | `command -v node` + version check >= 20 | Covers direct installs, Homebrew, system Node |
| 2 | nvm | `$NVM_DIR/nvm.sh` exists, source it, `nvm use default` | Must source nvm.sh first since it's a shell function |
| 3 | fnm | `command -v fnm`, `eval "$(fnm env)"` | fnm works in non-interactive shells natively |
| 4 | volta | `command -v volta`, `volta which node` | Volta shims are usually already on PATH, but check explicitly |
| 5 | Homebrew | `brew --prefix node@22` or `brew --prefix node` | Check if installed but not linked |
| 6 | Common paths | `/usr/local/bin/node`, `/opt/homebrew/bin/node` | Catch unlisted installs |

### Fallback by context

- **In `install.sh`**: Nothing found -> `brew install node` -> retry. No Homebrew either -> error with instructions.
- **In launcher wrapper** (launchd): Nothing found -> write error to log, exit 1. Never install at launch time.

### Version negotiation

If Node found but version < 20:
- **install.sh**: Warn, offer Homebrew install alongside, or suggest upgrading nvm/fnm default.
- **Launcher wrapper**: Skip to next option in chain. Stale Node shouldn't block if a newer one exists elsewhere.

## The `shooter` CLI

`bin/shooter.cjs` becomes the single user-facing interface.

### Commands

| Command | What it does |
|---------|-------------|
| `shooter start` | Start server in foreground (default if no args). Use `shooter autostart on` for background/persistent mode via launchd. |
| `shooter stop` | Find running server (by PID file or launchctl), send SIGTERM |
| `shooter restart` | `stop` then `start` |
| `shooter status` | Show: running/stopped, PID, port, uptime, Node version, last health check |
| `shooter setup` | Interactive wizard (existing `setup.cjs`) |
| `shooter update` | Check for dirty state first (`git status --porcelain`), stash if needed, `git pull --rebase` (abort on conflict), `pnpm install` + `pnpm build`, pop stash, restart if was running |
| `shooter logs` | Tail launchd logs (`~/Library/Logs/Shooter/`). `--follow` for live tail. |
| `shooter autostart on` | Generate launcher.sh + plist, `launchctl bootstrap` |
| `shooter autostart off` | `launchctl bootout`, remove plist |
| `shooter version` | Version + Node version + install path |
| `shooter help` | List all commands |

### PID tracking

- `shooter start` checks if `~/.shooter/shooter.pid` exists AND the process is alive (`kill -0 $pid`) before starting; refuses if already running
- Writes PID + start timestamp atomically (write to `.pid.tmp`, then rename)
- Format: `{"pid": 12345, "startedAt": "2026-03-26T10:00:00Z", "port": 3000}`
- `shooter stop` reads PID, verifies the process is a node/tsx process (via `ps -p $pid -o command=`) before sending SIGTERM; removes PID file after stop
- If autostart is on, `launchctl list | grep com.shooter.server` is the authority over the PID file
- `shooter status` derives uptime from `startedAt` in the PID file

### pnpm resolution (internal)

The CLI resolves pnpm via a cascading fallback:
1. `corepack pnpm` (preferred ‚Äî works if `corepack enable` succeeded during install)
2. `~/.shooter/repo/node_modules/.bin/pnpm` (local fallback ‚Äî always available after first install)
3. Global `pnpm` on PATH (last resort)

**Note on corepack:** `corepack enable` writes shims to the same directory as Node's binary. If that directory requires elevated permissions (e.g. `/usr/local/bin` on Intel Macs), it will fail. The installer handles this gracefully: try `corepack enable`, if it fails, fall back to `npm install -g pnpm` (which goes to the user's npm prefix), or skip entirely and use `npx pnpm` for the initial install. After `pnpm install` completes, `node_modules/.bin/pnpm` is always available as the reliable fallback.

All pnpm calls are internal. User output shows human-readable progress:

```
$ shooter update
[*] Pulling latest changes...
[*] Installing dependencies...
[*] Building...
[+] Updated to v1.2.3
[*] Restarting server...
[+] Server running on http://localhost:3000
```

### Error experience

Clear messages with actionable fixes:

```
[-] Port 3000 is already in use.
    Try: Set PORT=3001 in ~/.shooter/.env
    Or:  lsof -i :3000  (to find what's using it)
```

## Launcher Wrapper Script

Generated by `shooter autostart on` at `~/.shooter/launcher.sh`. Machine-specific, never committed to the repo.

### .env compatibility note

The `.env` file is sourced by both `dotenv` (in server.ts) and `bash source` (in launcher.sh). Multi-line values (APNS_KEY, FCM_PRIVATE_KEY) are stored with literal `\n` inside double quotes ‚Äî compatible with both parsers. `setup.cjs` must ensure this format.

### Structure

```bash
#!/bin/bash
# Auto-generated by shooter autostart ‚Äî do not edit manually
# Regenerate with: shooter autostart on

SHOOTER_HOME="$HOME/.shooter"
SHOOTER_REPO="$SHOOTER_HOME/repo"

# ‚îÄ‚îÄ Source environment ‚îÄ‚îÄ
set -a
[ -f "$SHOOTER_HOME/.env" ] && source "$SHOOTER_HOME/.env"
set +a

cd "$SHOOTER_REPO"

# ‚îÄ‚îÄ Fast path: hardcoded Node (set at install time) ‚îÄ‚îÄ
NODE_BIN="/opt/homebrew/bin/node"  # actual path detected at generation time

if [ -x "$NODE_BIN" ]; then
    NODE_VERSION=$("$NODE_BIN" --version 2>/dev/null | sed 's/v//' | cut -d. -f1)
    if [ -n "$NODE_VERSION" ] && [ "$NODE_VERSION" -ge 20 ] 2>/dev/null; then
        export PATH="$(dirname "$NODE_BIN"):$SHOOTER_REPO/node_modules/.bin:$PATH"
        exec "$SHOOTER_REPO/node_modules/.bin/tsx" "$SHOOTER_REPO/server.ts"
    else
        echo "$(date): WARN ‚Äî Node at $NODE_BIN is version $NODE_VERSION (need 20+), trying fallback chain..." >&2
    fi
fi

# ‚îÄ‚îÄ Self-healing fallback: detection chain ‚îÄ‚îÄ
# Each candidate must pass version >= 20 check before being accepted
check_node_version() {
    local candidate="$1"
    local ver
    ver=$("$candidate" --version 2>/dev/null | sed 's/v//' | cut -d. -f1)
    [ -n "$ver" ] && [ "$ver" -ge 20 ] 2>/dev/null
}

find_node() {
    local candidate
    # 1. PATH
    candidate="$(command -v node 2>/dev/null)"
    [ -n "$candidate" ] && check_node_version "$candidate" && { echo "$candidate"; return; }
    # 2. nvm
    if [ -s "$HOME/.nvm/nvm.sh" ]; then
        . "$HOME/.nvm/nvm.sh"
        candidate="$(command -v node 2>/dev/null)"
        [ -n "$candidate" ] && check_node_version "$candidate" && { echo "$candidate"; return; }
    fi
    # 3. fnm
    if command -v fnm >/dev/null 2>&1; then
        eval "$(fnm env)"
        candidate="$(command -v node 2>/dev/null)"
        [ -n "$candidate" ] && check_node_version "$candidate" && { echo "$candidate"; return; }
    fi
    # 4. volta
    [ -x "$HOME/.volta/bin/node" ] && check_node_version "$HOME/.volta/bin/node" && { echo "$HOME/.volta/bin/node"; return; }
    # 5. Homebrew paths
    [ -x "/opt/homebrew/bin/node" ] && check_node_version "/opt/homebrew/bin/node" && { echo "/opt/homebrew/bin/node"; return; }
    [ -x "/usr/local/bin/node" ] && check_node_version "/usr/local/bin/node" && { echo "/usr/local/bin/node"; return; }
}

FOUND_NODE="$(find_node)"
if [ -z "$FOUND_NODE" ]; then
    echo "$(date): FATAL ‚Äî No Node.js >= 20 found. Run 'shooter autostart on' to regenerate." >&2
    exit 1
fi

export PATH="$(dirname "$FOUND_NODE"):$SHOOTER_REPO/node_modules/.bin:$PATH"
exec "$SHOOTER_REPO/node_modules/.bin/tsx" "$SHOOTER_REPO/server.ts"
```

### Launchd plist

The plist at `~/Library/LaunchAgents/com.shooter.server.plist` calls the wrapper:

```xml
<key>ProgramArguments</key>
<array>
    <string>/bin/bash</string>
    <string>/Users/username/.shooter/launcher.sh</string>  <!-- actual $HOME expanded at generation time -->
</array>
```

No PATH or environment variables in the plist ‚Äî the wrapper handles everything.

### Properties
- Fast path first (~0ms overhead if hardcoded path works)
- Self-healing if Node moves (upgrade, version manager switch)
- Logs errors with timestamp for `shooter logs`
- Regenerated any time user runs `shooter autostart on`

### Log management

Logs go to `~/Library/Logs/Shooter/stdout.log` and `stderr.log`. To prevent unbounded growth:
- `shooter autostart on` installs a `newsyslog` config at `/etc/newsyslog.d/shooter.conf` (if writable) or the launcher itself truncates logs beyond 50MB on startup
- `shooter logs --clear` truncates both log files
- `shooter status` shows current log file sizes

## Corporate / Edge Case Handling

### Network

| Scenario | Handling |
|----------|----------|
| HTTP proxy | `install.sh` respects `$HTTP_PROXY`/`$HTTPS_PROXY`/`$NO_PROXY` ‚Äî git, npm, brew all honor these. No special code. |
| Private npm registry | `npx` path works with user's `.npmrc`. `curl\|sh` uses git clone, bypasses registry. |
| Restricted git access | If `git clone` fails: "Can't reach github.com. Try: `npx @juspay/shooter setup` (uses npm registry instead)." |

### Permissions

| Scenario | Handling |
|----------|----------|
| No sudo | Nothing requires sudo. `corepack enable` may fail if Node's bin dir is restricted ‚Äî installer falls back to `npm install -g pnpm` (uses user's npm prefix) or `node_modules/.bin/pnpm`. CLI symlinks to `~/.local/bin/` or `~/bin/`. |
| Read-only /usr/local | Homebrew on Apple Silicon uses `/opt/homebrew` (user-owned). We don't write to /usr/local. |
| Home directory quotas | `~/.shooter` is ~200MB total. `shooter status` shows install size. |

### Node version conflicts

| Scenario | Handling |
|----------|----------|
| System Node is 16/18 | Detection chain finds it, sees < 20, moves to next. Doesn't touch system Node. |
| nvm default is old | Warn: "nvm default is Node 18. Shooter needs 20+. Run `nvm install 22` or we'll use Homebrew." |
| Multiple Nodes on PATH | Use first >= 20 found. `shooter status` shows which Node is active. |

### Upgrade / migration

| Scenario | Handling |
|----------|----------|
| Existing install | `install.sh` already has update/fresh/abort flow. Preserved. |
| Node upgrade | Launcher self-heals. Or `shooter autostart on` to regenerate. |
| Breaking version | `shooter update` pulls + rebuilds. `setup.cjs` detects .env version, prompts for migration. |
